## Abstract

Jolt Atlas is a zero-knowledge machine learning (zkML) framework that applies Jolt's lookup-argument architecture directly to ONNX tensor operations, enabling cryptographic proofs that a model inference was executed correctly without revealing the model weights, input data, or intermediate activations. The framework uses sumcheck-based lookup arguments optimized for non-linear functions, neural teleportation for table size reduction, and the BlindFold technique for full zero-knowledge. It supports streaming proofs in memory-constrained environments and succinctly verifiable proofs without specialized hardware. From a cognitive warfare perspective, Jolt Atlas represents a protocol-level mechanism for establishing or attacking epistemic trust in AI-mediated outputs, with direct applications to both defensive attestation and offensive deception pipelines.

## Cognitive Warfare Analysis

### Threat Model Relevance

The core manipulation surface in AI-assisted decision-making is the inference step: a human or automated system receives a model output and acts on it. Currently, nothing about that output is verifiable. An adversary who compromises an inference pipeline can substitute outputs, perturb logits, or serve a different model entirely without any observable signature. Jolt Atlas closes this gap by generating a cryptographic proof that a specific model (committed to by its parameters) produced a specific output from a specific input. This proof can be verified by a third party in milliseconds without access to the model or input.

### Defensive Applications

**1. Chain-of-custody attestation for intelligence products.** Analytical products generated by AI (threat assessments, targeting recommendations, signal classifications) can be bound to a verifiable model identity. A recipient can confirm that the output was produced by the authorized model version, not a substituted or fine-tuned adversary variant. This is directly relevant to agentic systems operating in denied or contested environments.

**2. Tamper detection in human-AI teaming.** In cognitive teaming architectures, a human operator trusts AI-generated context windows, summaries, or options packages. If those are produced by a zkML-attested model, any post-inference manipulation of the output (injection, truncation, reordering) is detectable because the proof covers the full output tensor.

**3. AI memory integrity.** The companion work cited by the authors explicitly addresses "trustless AI context" (AI memory). In agentic systems, poisoned memory is a primary cognitive attack vector: an adversary writes false episodic records into an agent's context store, which then propagates into future reasoning. zkML attestation on retrieval and summarization steps constrains this attack surface by requiring that stored and retrieved context be bound to a verifiable inference trace.

**4. Model identity verification under adversarial fine-tuning.** A known-safe model can be parameter-committed before deployment. Any downstream fine-tuning (e.g., from RLHF poisoning, backdoor injection, or adversarial data supply) changes the commitment hash. zkML proofs from the altered model will fail to verify against the original commitment, providing a tripwire.

### Offensive Applications

**1. Proof forgery as a trust weapon.** If an adversary can subvert the commitment mechanism (e.g., by compromising the trusted setup or the commitment infrastructure), they can generate valid-seeming proofs for a malicious model. This is analogous to a fraudulent certificate authority in TLS: the verification step becomes a false confidence signal, potentially inducing higher trust in adversary-controlled outputs than in unattested legitimate ones. The attack does not require breaking the cryptography; it requires controlling the enrollment step.

**2. Selective proof presentation.** A zkML proof covers a specific input-output pair. An adversary in control of the inference server can selectively generate proofs for outputs that support a desired narrative while withholding or refusing to generate proofs for contradictory outputs. To a downstream verifier, the attested outputs appear trustworthy, while the suppressed outputs are simply absent. This is a cryptographically-assisted selection bias attack.

**3. Proof as covert channel.** The proof artifacts themselves (Fiat-Shamir transcripts, sumcheck polynomials) have structure that could carry steganographic payload. This has not been studied in the literature but is a non-trivial attack surface for exfiltration from air-gapped or policy-controlled inference environments.

**4. Weaponized compliance theater.** In regulatory or procurement contexts where zkML attestation is mandated as a safety check, an adversary can use a compliant zkML pipeline to certify a model that is behaviorally safe on the audit distribution but adversarially tuned on out-of-distribution inputs. The proof is valid; the safety guarantee is not. This exploits the gap between formal verification (what the proof covers) and behavioral robustness (what the auditor cares about).

### Key Technical Constraints

Jolt Atlas proofs cover inference correctness, not inference alignment or output truthfulness. A model can be provably executing its weights and still be producing false, manipulated, or harmful outputs. Defenders must not conflate "attested inference" with "trustworthy output." The threat model for cognitive manipulation does not require corrupting the cryptographic layer; it requires controlling the model or the input distribution before the proof is generated.
